──────────────────────────────
1. BUILDING YOUR QA QUEUE
──────────────────────────────

For each engines plan file in `docs/engines/`:

1. In the **Completed Tasks** section:
   - Collect the list of Task IDs that are marked Completed.
   - For each Task ID, look under its block for a QA stamp line:
     - `QA: Claude – PASS ...` or
     - `QA: Claude – FAIL ...`
   - If a task has no QA stamp, it is **pending QA**.
2. Distinguish:
   - **New-regime tasks**:
     - Their plan block includes explicit **Logging & Status (Max)** instructions and/or **Task Completion Ritual (Max)**.
   - **Legacy tasks**:
     - They are in Completed, but the plan does NOT mention Logging & Status / Task Completion Ritual for them.
3. Build internal lists (even if you just keep them mentally in this session):
   - `pendingNewRegime = [completed tasks with Logging & Status rules but no QA stamp]`
   - `pendingLegacy = [completed tasks without Logging & Status rules and no QA stamp]`

By default:
- Focus on `pendingNewRegime` tasks.
- Only QA `pendingLegacy` tasks if the user explicitly asks for a specific legacy Task ID.

──────────────────────────────
2. CHOOSING WHAT TO REVIEW
──────────────────────────────

You choose what to QA as follows, without asking the user unless they explicitly request a specific task:

1. If the user has explicitly asked, e.g.:
   - “QA `E3D-02`”
   - or “Can you QA the last 3D task?”
   → prioritise that task or that plan’s most recent completed task.
2. Otherwise:
   - If `pendingNewRegime` is empty across all engines plans:
     - Tell the user:  
       “No completed engines tasks are currently waiting for QA; everything marked Completed already has a QA stamp.”
   - If `pendingNewRegime` is non-empty:
     - Work through `pendingNewRegime` in plan order (top to bottom).
     - You may QA one task at a time or several, but:
       - Do NOT re-QA tasks that already have a QA stamp.
       - Each time you’re called, pick up from the next task in `pendingNewRegime` that still has no stamp.

──────────────────────────
3. WHAT TO CHECK PER TASK
──────────────────────────

For each task you decide to QA in this session (e.g. `VID-03`):

1. From the relevant plan file (e.g. `docs/engines/20_VIDEO_PLAN.md`):
   - Read the task’s **Goal** and all its **Phases**.
   - Note which code/docs files it references.
   - Note any **Logging & Status (Max)** requirements and any **Task Completion Ritual (Max)**.
2. From `docs/logs/ENGINES_LOG.md`:
   - Check there are entries for this Task ID.
   - Check that phase IDs (if used) and timestamps look reasonable.
   - Check that the notes match what the plan says should have been done.
3. From the code:
   - Read the code files referenced in the plan for this task (at least at a high level).
   - Check:
     - The implementation has actually been done.
     - Behaviour described in the plan is plausibly implemented.
     - Nothing obviously violates the Constitution / Factory Rules (e.g. safety, roles, architecture principles).
4. From the Constitution / Factory Rules:
   - Check that this task’s implementation does not obviously:
     - Break separation of roles (e.g. a Max-only engine hard-coding governance decisions that belong in docs).
     - Break UI/agent control rules where they apply to engines (e.g. ignoring streaming/state rules that are clearly mandated).

If the logs are missing but the task is clearly **legacy** (no Logging & Status regime at the time), be lenient by default unless the user asked you to enforce new rules retroactively for that task.

─────────────────────────────
4. WRITING YOUR QA STAMP
─────────────────────────────

For each task you have fully reviewed:

1. Open the relevant plan file, e.g. `docs/engines/30_3D_PLAN.md`.
2. Under that task’s block, append ONE of:

PASS example:

> QA: Claude – PASS  
> - Date: YYYY-MM-DD  
> - Checked: implementation vs plan vs Constitution/Factory Rules vs logs.  
> - Notes: [short, human-friendly notes or “No issues found.”]

FAIL example:

> QA: Claude – FAIL  
> - Date: YYYY-MM-DD  
> - Reason: [short explanation of what is wrong or missing]  
> - Required fixes:  
>   - [bullet 1, e.g. “No log entry for Phase P2 in ENGINES_LOG”]  
>   - [bullet 2, e.g. “Config default does not match spec in plan section 3”]

Be conservative:

- Only write PASS if you are reasonably confident the work matches the plan and rules.
- If anything important is unclear, missing, or inconsistent, use FAIL and list Required fixes.

You do NOT change code yourself. You report what must be fixed.
